# Local-To-Production Integration Plan

## الهدف

توحيد النسخة المحلية المتقدمة من `Tiger.mail / EmailArray Outlook` مع النسخة المنشورة على:

- `https://techno-grp--com.w.emailarray.com/#email`

بحيث تصبح بيئة الإنتاج هي النسخة الرسمية التي تعمل بنفس:

- منطق الموافقات `Approval Workflow`
- تحليل المخاطر `AI Risk Analysis`
- إشعارات Telegram
- مزامنة البريد `POP3 / IMAP / Graph`
- الأرشفة وسجل التدقيق `Audit Trail`

## الوضع الحالي

يوجد غالبا فرق بين:

- النسخة المحلية: تحتوي آخر التعديلات والمنطق الأحدث
- النسخة المنشورة: قد تعمل بكود أقدم أو ببيانات أقدم أو بإعدادات مدير/موظف غير متطابقة

وهذا يفسر أعراض مثل:

- اختفاء `Pending Approvals` عند المدير
- اختلاف حساب الأدمن أو صلاحياته
- إرسال الرسالة مباشرة بدل حفظها كـ `Pending Approval`

## الهدف المعماري النهائي

الهيكل الصحيح للإنتاج يجب أن يكون كالتالي:

1. `Frontend`
   الواجهة المنشورة على `emailarray.com`

2. `Backend`
   نفس خادم `Node.js / Express` الموجود في هذا المشروع

3. `Database`
   قاعدة بيانات إنتاجية مركزية واحدة

4. `Uploads Storage`
   مجلد مرفقات ثابت على السيرفر

5. `Background Jobs`
   الجلب التلقائي كل 5 دقائق

6. `Notifications`
   Telegram + AI analysis

## التوصية الأساسية

استخدم في الإنتاج:

- `PostgreSQL` كقاعدة البيانات الرسمية
- نفس الـ backend الحالي
- نفس الواجهة الحالية بعد `build`

ولا تترك بيئتين منفصلتين تعملان على بيانات مختلفة.

## لماذا PostgreSQL في الإنتاج

البرنامج لم يعد مجرد واجهة تجريبية، بل أصبح يحتوي على:

- مستخدمين فعليين
- موافقات متعددة المراحل
- مزامنة بريد ومرفقات
- تقارير وتحليلات
- سجل تدقيق

ولهذا `PostgreSQL` أفضل من الاعتماد على `pg-mem` في بيئة الإنتاج.

## مسار الدمج المقترح

### المرحلة 1: تثبيت بيئة الإنتاج

- تجهيز سيرفر Linux أو Windows Server
- تثبيت `Node.js`
- تثبيت `npm`
- تجهيز `PostgreSQL`
- تجهيز Proxy أو Web Server إذا لزم

### المرحلة 2: توحيد الإعدادات

على السيرفر يجب إعداد:

- `PORT`
- `DATABASE_URL`
- `JWT_SECRET`
- `SYNC_API_KEY`
- `APP_BASE_URL`
- `TELEGRAM_BOT_TOKEN`
- `OPENAI_API_KEY` أو `GEMINI_API_KEY`

### المرحلة 3: نقل الكود المحلي إلى الإنتاج

- رفع المشروع من جهازك إلى السيرفر
- تشغيل:
  - `npm install`
  - `npm run build`
  - `npm start` أو `pm2 restart`

### المرحلة 4: توحيد قاعدة البيانات

يجب نقل البيانات المهمة من المحلي إلى الإنتاج:

- `users`
- `user_mail_settings`
- `emails`
- `attachments`
- `approval_logs`
- `email_trail`
- `recommendations`
- `reminders`
- `email_archives`

### المرحلة 5: تفعيل قواعد العمل

التأكد أن بيئة الإنتاج تحافظ على نفس المنطق:

- أي مستخدم غير `Admin` لا يرسل خارجيًا مباشرة
- وجود `Direct Manager` يفرض مسار الموافقة
- المدير يرى `Pending Approvals`
- الموظف لا يطالع صندوق المدير
- الرسائل تمر عبر AI قبل الاعتماد
- كل رسالة تحمل `Serial ID`

## حل مشكلة Pending Approvals على الإنتاج

إذا اختفت `Pending Approvals` عند المدير فالأسباب الأقرب هي:

1. نسخة الإنتاج لم تُحدّث بعد بآخر كود
2. السيرفر لم يُعد تشغيله بعد النشر
3. الموظف غير مربوط بمدير مباشر في قاعدة بيانات الإنتاج
4. الرسالة لا تُحفظ بحالة `pending`
5. المدير يدخل بحساب مختلف عن المدير المربوط على الرسالة

### شرط ظهور الرسالة عند المدير

يجب أن تكون الرسالة في قاعدة البيانات الإنتاجية بهذه الصورة:

- `approval_status = 'pending'`
- `assigned_manager_id = manager.id`

أو:

- `employee.manager_id = manager.id`

- تصنيف المرفقات
- معاينات أفضل
- فلاتر حسب النوع والحجم والخطورة

### 7. Search Upgrade

- بحث موحد عبر:
  - السيريال
  - الموضوع
  - المرسل
  - المستلم
